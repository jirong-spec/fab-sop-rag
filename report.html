四階段 Guardrail 設計

# 名稱 類型 觸發條件 實作方式 延遲
1 Injection Detection Regex 問題含 prompt injection 關鍵字（英/中） 20 條正規表達式 pattern matching，含 "ignore previous instructions"、"扮演"、"你現在是" 等英/中攻擊向量 ~0 ms
2 Topic Filter Regex LLM 問題不屬於晶圓廠 SOP 7 大類別 快速通道（無 vLLM）：含 SOP ID 或 CamelCase 實體（如 SOP_Etch_001、EtchStation）→ 直接放行
SOP_[A-Za-z0-9_]+|(?:[A-Z][a-z][A-Za-z0-9]*){2,}

LLM-as-judge（~600 ms）：無 entity code → 送 vLLM 判斷，回傳 {"relevant": bool, "reason": str}
你是一位晶圓廠 SOP 文件庫的主題過濾助理。 你的任務是判斷工程師的問題是否屬於「晶圓廠 SOP 知識庫」的查詢範疇。 本知識庫的查詢範疇涵蓋以下七類： 1. 製程異常處置（壓力異常、溫度超標、真空度不足⋯） 2. SOP 操作步驟與流程（排查程序、前置確認、interlock 條件） 3. 設備與機台狀態條件（pump 狀態、腔體 interlock、感測器閾值） 4. 步驟依賴關係（PRECONDITION、NEXT_STEP、DEPENDS_ON） 5. 製程條件與 recipe 參數（溫度、壓力、氣體流量） 6. 良率異常與排查流程（缺陷模式、Cpk 異常） 7. 跨文件設備依賴關係 不屬於範疇：一般程式設計、數學、市場行情、財務、人事管理 問題：{question} 請僅回傳 JSON： {"relevant": true, "reason": "簡短說明（20字以內）"} {"relevant": false, "reason": "簡短說明（20字以內）"}

3 Evidence Sufficiency Code Neo4j 回傳 0 條 triple 純 Python，無 LLM call：len(triples) < min_count → block
min_count = 1（保守設定：有任何 triple 就放行）

設計理由：語意品質由 Reranker 與 Guard 4 負責；此處只防「完全無圖譜證據」就送 LLM 的最惡劣幻覺情況。 		~0 ms
4 Fact Grounding LLM 答案包含圖譜以外的推論或幻覺 LLM-as-judge：將答案與 model_triples 送入 vLLM，逐項核對 SOP 步驟、設備狀態、前置條件是否有圖譜依據。
失敗 → 不 block，降級為 answered_with_warning（confidence 0.5）		 ~1100 ms
設計邏輯
越早的 guard 越便宜 Guard 1 純 Regex（~0 ms）→ Guard 2 Regex pre-check → Guard 2 LLM（~600 ms）→ Guard 3 純計數（~0 ms）→ Guard 4 LLM（~1100 ms）。高成本的 LLM 判斷放最後，讓 Regex 先快速過濾明顯攻擊與明確 in-domain 問題。
越晚的 guard 越保守 Guard 1–3 失敗 → 直接 status=blocked，答案不出去。Guard 4 失敗 → 降級為 answered_with_warning（confidence 0.5）。Guard 4 已是最後防線，靜默丟棄答案不如明確警告讓工程師自行確認。
Guard 3 故意輕量 只問「有沒有 triple」，不問「triple 有沒有語意相關」。語意品質已由 Reranker 排序處理，Guard 4 負責事後驗證。若用 LLM 做語意判斷，7B 模型在 40+ 條雜訊 triple 中容易漏判（needle-in-haystack），反增 false negative 率。
Guard 2 的雙軌設計 含 SOP entity code 的問題由 Regex 直接放行，避免小模型誤判合法 SOP 查詢為 off-topic。其餘問題呼叫 LLM judge，fallback policy 預設 lenient（LLM 服務失敗時放行），避免 LLM 故障導致全面 block。

兩個 pipeline 的 Guardrail 成本幾乎相同（guard_topic 在有 SOP entity code 時為 0 ms）。 Graph RAG 多出的 ~251 ms overhead 來自：

  • Neo4j 圖遍歷（2-hop Cypher 查詢）：+~100 ms
  • Reranker 批次 embedding（40–62 條 triple，GPU）：+~85 ms
  • Generation prompt 長度（graph triple prompt token 略多）：+~70 ms
  • guard_grounding context（model_triples ~15–19 條 vs Qdrant 6 個片段）：接近相同
06

核心技術決策與優化歷程

問題 根本原因 解法
q10 A❌（LogAndClearAlarm 遺失） threshold = 79 × 0.50 = 39.5（float），chain 最後一條 triple 得分 39（int），39 ≥ 39.5 = False 改為 int(scored_all[0][0] * 0.50)，floor 避免浮點邊界截斷
步驟順序問題（NEXT_STEP chain 遺失） MiniLM 對「步驟順序」語意匹配傾向 DEFINED_IN / TRIGGERS_SOP，NEXT_STEP triple 得分低於 threshold 偵測到問題含 SOP ID + 「步驟 / 順序 / 流程」時，強制補入該 SOP 的所有 NEXT_STEP triple
cross-SOP 步驟洩漏（多回答） Dynamic cap 從 62 條 triple 中選出的集合包含其他 SOP 的步驟節點，7B 模型全部列出 兩段式 SOP filter：Pass 1 找出 foreign steps，Pass 2 移除所有 foreign triple
guard_grounding 過慢 傳入全部 evidence_triples（40–62 條）而非 model_triples（15–19 條），LLM prefill 過長 改為 guard_grounding(answer, model_triples)，節省 ~365 ms
Embedding 在 CPU 跑（GPU 閒置） API container 未設定 nvidia deploy；PyTorch 安裝 CUDA 13.0 但 driver 只支援 12.4 Dockerfile 明確安裝 torch==2.5.1+cu124；docker-compose 加 GPU device reservation
q01 冷啟動異常（~11 s → ~3.3 s） pre-warm 只載入 _get_embeddings()，_get_reranker_embeddings() 仍為冷啟動；僅載入模型物件不觸發 GPU kernel 初始化，需實際呼叫 embed_query() lifespan 與 eval pre-warm 均改為對兩個 embedding model 呼叫 .embed_query("warmup")
向量庫 Chroma → Qdrant 原 in-process Chroma（persist_directory）不利並發/擴展；改用獨立向量 server 讀寫統一 LangChain QdrantVectorStore（payload schema 一致）；ingest force_recreate 避免刪檔後殘留孤兒向量。Graph RAG 不變，Vector baseline 54%→62%
圖遍歷 LIMIT 200 截斷（大圖漏邊） RETURN p LIMIT 200 限制的是「路徑數」，2-hop 路徑爆炸會在撈到關鍵邊前就截斷 改 distinct-edge 遍歷（收 hop 內不重複邊，LIMIT 500）；實測 vs undirected 準確率打平且消除截斷，有向（directed）反而退步 → 已否決
大圖 prompt 撐爆 context（HTTP 400） 動態 cap 只限 triple 數（max 100），dense 子圖序列化後超過 4096 token，生成整題失敗 token-aware 裁切：依 llm_max_model_len − 完成 − margin 預算保留高分 triple，prompt 絕不溢出 context
低相似度邊被 rerank 砍掉（recall 掉） REQUIRES_STATUS / PRECONDITION / INTERLOCK 邊只有 CamelCase ID，對中文問題 embedding 相似度趨近零 為缺 description 的邊型合成中文 gloss（比照 NEXT_STEP/DEPENDS_ON enrichment）；10-SOP step_requires_status recall 50%→100%

08

結論

Graph RAG 相較於傳統 Vector RAG，在晶圓廠 SOP 結構化查詢場景下有顯著優勢：

100% Graph RAG keyword 100%、嚴謹 held-out 評估亦 100%、10-SOP 壓測 96%；Vector RAG 僅 62.5%。差距集中於步驟順序、Interlock 條件、multi-hop 跨文件查詢等需要圖結構推理的問題。
+251ms 額外延遲僅 +251 ms（約 6%）。主因是 Neo4j graph traversal 與 reranker embedding，不是 LLM 呼叫次數差異。
0 幻覺 Guard 4 事後驗證 + model_triples 精準傳遞，確保答案完全基於 SOP 圖譜，無推論或補充內容越界。

Vector RAG 在純語意查詢（設備狀態、簡單依賴）準確率相近，但在需要沿 NEXT_STEP / DEPENDS_ON 鏈推理的問題上完全失敗。 Graph RAG 的結構化檢索是解決半導體製造 SOP 多步驟依賴查詢的根本性優勢。
